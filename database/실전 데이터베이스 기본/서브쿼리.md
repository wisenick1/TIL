상관 서브쿼리
--
메인 쿼리의 값이랑 서브쿼리의 값이 상관이 있을 때!

where 절에서 IN을 사용할 경우 데이터가 아주 많으면 문제가 될 수 있다 -> 먼저 EXISTS를 고려해보자

상관 서브쿼리는 실행 횟수가 굉장히 많아질 수 있어 성능 저하로 이어질 수 있다. 따라서 JOIN을 먼저 고려하고, 고려하는 방향으로~

인라인 뷰
--
테이블 서브쿼리

하나의 데이터 집합을 만들어 놓고, 거기서 SELECT를 하는 방식


사고의 흐름 자체가 어떤 문제가 주어지면 어떤 순서로 해결할지 쪼개보고 그에 따라 SQL을 작성해서 필요 시 서브쿼리, 조인 등을 활용하면 된다고 생각한다.

근데 문제 상황에서 어떤 상황에 상관 서브쿼리를 쓸지, 인라인 뷰를 쓸지는 짜다보면 보일지도 모르겠는데 한번에 이렇게 해야겠다는 기준은 없는 듯 -> 아직은 경험을 바탕으로 쌓아가는 것이 맞다고 생각

기준을 조금 잡자면 where 절에는 특정 조건을 맞춘 애들을 출력하겠다.

from은 데이터를 가공해서 이를 바탕으로 재활용 하여 일치하는 데이터들을 출력하겠다.

select는 각 행에 부가 정보를 출력하겠다 의 느낌이다.

이걸 바탕으로 쿼리를 짜보면 더 빠르게 기준을 잡아갈 수 있을 듯
